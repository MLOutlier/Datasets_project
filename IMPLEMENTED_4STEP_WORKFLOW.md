# Реализованный 4-шаговый workflow (текущее состояние)

Документ фиксирует, что уже реализовано в системе, и как сейчас проходит процесс от загрузки данных до экспорта датасета.

## 1. Что реализовано сейчас

### Этап 1. Выделение интервалов видео с объектом (исполнители)

- Добавлена доменная сущность `VideoInterval` со статусами:
  - `draft`
  - `approved`
  - `rejected`
- Интервалы привязаны к проекту и видео-ассету.
- Поддерживаются источники интервалов:
  - `auto` (автогенерация)
  - `manual` (ручное добавление/изменение)
- Реализованы assignment-driven задачи по video chunks:
  - `GET /api/annotator/interval-chunks/queue/`
  - `POST /api/annotator/interval-chunks/{assignment_id}/submit/`

### Этап 2. Валидация интервалов (другие исполнители)

- Реализована validator-driven валидация:
  - отдельные задания валидации интервалов
  - правило `validator != interval author`
  - минимум 3 валидатора на интервал
  - агрегация голосов в final decision (`approved/rejected`)
- API:
  - `GET /api/annotator/interval-validations/queue/`
  - `POST /api/annotator/interval-validations/{assignment_id}/submit/`

### Этап 3. Выделение объекта на изображении (bbox)

- Формирование задач bbox связано только с подтвержденными интервалами:
  - при финализации импорта (`finalize`) для видео в `work_items` попадают только кадры из `approved` интервалов.
- Сохранен текущий поток аннотатора:
  - работа по assignment
  - `submit and next`
  - batch-контекст.
- Предразметка больше не placeholder-пустышка:
  - теперь возвращается baseline bbox-кандидат с confidence.

### Этап 4. Валидация bbox (исполнители, без reviewer-ветки)

- Реализована очередь bbox validation для annotator:
  - отдельные validator assignments
  - правило `валидатор не может валидировать свой bbox`
  - пакетный формат: `20 real + 10 golden`
  - порог по golden: `>= 80%`
- API:
  - `GET /api/annotator/bbox-validations/queue/`
  - `POST /api/annotator/bbox-validations/{assignment_id}/submit/`

### Экспорт и надежность пайплайна

- Расширены форматы экспорта:
  - `COCO`
  - `YOLO`
  - `VOC`
  - `CSV`
  - `both` (комбинированный payload)
- Поддержан экспорт архива `zip`.
- Добавлена диагностика `ffmpeg` и вывод статуса в preview импорта.

## 2. Как сейчас проходит процесс от загрузки до выгрузки

Ниже описан целевой рабочий путь в текущей реализации.

### Шаг 0. Создание и настройка проекта

1. Заказчик создает проект CV (`project_type=cv`, `annotation_type=bbox`).
2. Настраивает классы (`label_schema`), параметры quality и правила workflow.
3. При необходимости загружает инструкцию для исполнителей.

### Шаг 1. Загрузка медиа и первичная обработка

1. Заказчик открывает страницу проекта.
2. Загружает изображения/видео через import.
3. Для видео выполняется извлечение кадров (через `ffmpeg`).
4. Система формирует preview:
   - число обработанных/ошибочных ассетов
   - число кадров
   - cleanup-показатели
   - статус доступности `ffmpeg`.

### Шаг 2. Интервалы видео (исполнители)

1. Система режет видео на чанки.
2. Исполнители получают задания по чанкам в очереди `interval-chunks`.
3. Исполнитель отмечает интервалы с объектом внутри чанка и отправляет результат.

### Шаг 3. Валидация интервалов исполнителями

1. Другие исполнители получают interval-validation задачи.
2. Один и тот же интервал проверяют минимум 3 валидатора.
3. При достижении кворума система выставляет final статус интервала.
4. После `approved` интервала кадры сразу (rolling flow) попадают в bbox этап.

### Шаг 4. Finalize импорта и генерация задач

1. Заказчик возвращается в проект и нажимает `Finalize import`.
2. Система создает `work_items` и `assignments`:
   - для изображений — как обычно,
   - для видео — только из кадров, попавших в `approved` интервалы.

### Шаг 5. BBox-разметка аннотаторами

1. Аннотатор открывает свой проект и assignment.
2. Видит кадр, инструкцию, классы и (если есть) pre-annotation.
3. Рисует/редактирует bbox, сохраняет draft или отправляет final.
4. После submit система ведет аннотатора на следующий assignment.

### Шаг 6. Автоматическая оценка качества по agreement

1. После достаточного числа финальных аннотаций по кадру считается agreement.
2. Если agreement >= порога:
   - `work_item` принимается,
   - формируется финальная аннотация.
3. Если agreement < порога:
   - `work_item` переводится в review-поток,
   - создается `ReviewRecord`.

### Шаг 7. Валидация bbox исполнителями

- Исполнители получают validator assignments для bbox.
- Каждый пакет содержит:
  - 20 реальных кадров/вопросов
  - 10 контрольных (golden)
- Если golden score < 80% — пакет отклоняется.
- Если >= 80% — решения применяются, проблемные кадры уходят на доработку.

### Шаг 8. Экспорт датасета

1. Заказчик выбирает формат экспорта:
   - `coco`, `yolo`, `voc`, `csv`, `both`.
2. Получает payload экспорта через API или скачивает `zip`.
3. В выгрузку включается quality report и соответствующие секции форматов.

## 3. Технический итог

Текущая реализация поддерживает executor-driven 4-шаговый цикл:

1. Интервалы (auto+manual),
2. Валидация интервалов,
3. BBox-аннотация,
4. BBox-валидация исполнителями,

с дальнейшим экспортом датасета в несколько форматов.
